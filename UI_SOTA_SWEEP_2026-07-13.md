## The one thing

## Appendix A — all 40 confirmed findings

- **[typing] The typed line re-centers on every keystroke, so your own text drifts sideways as you write** — `src/components/Dashboard/Vigil.css:1293` (high/day)
  - Symptom: The line never sits still. Each character you type nudges everything you've already typed to the left and slides the caret right; when a suggestion appears or vanishes the whole line lurches again. It feels 'swimmy' and unanchored — the single biggest reason typing doesn't feel fluid.
  - Root cause: .tex-line-row is a flex row with justify-content:center inside the centered .tex-line-slot (line 1277), and the input is sized to its exact content (useLayoutEffect at Vigil.jsx:3014). Because the row is width-of-content and center-justified, the geometric center of (typed text + ghost) is pinned to the viewport center — so every width change (every keystroke, every ghost flip) re-centers the whole line and moves text that was already on screen. Nothing on the line has a stable horizontal origin.
  - Fix: Stop re-centering during an active typing burst. Keep the composed line horizontally frozen while keys are flowing and ease it (var(--tex-t3)) back to true-center only after a short pause in input — so short/finished questions still read centered (stark-white aesthetic preserved) but the text you are actively typing holds its position. Alternative surgical form: anchor the caret to a fixed x and let earlier text extend/scroll left of it. No capability removed; the ghost and centered look stay.
- **[typing] Ghost suggestion is part of the centered row, so it shoves your text sideways and pops in with no fade** — `src/components/Dashboard/Vigil.jsx:4232` (high/hours)
  - Symptom: The instant a completion appears (or disappears when the fragment turns ambiguous) your already-typed text jerks left/right, and the grey suggestion snaps in and out abruptly rather than surfacing softly. Typing toward an agent name feels jittery instead of assisted.
  - Root cause: The ghost is a flex sibling <span className="tex-ghost"> rendered conditionally ({typed !== null && ghost && ...}) inside the same center-justified row, so its width counts toward the centering math and its appearance/removal re-centers the typed text (compounds finding #1). Its CSS (.tex-ghost, Vigil.css:1352) has no transition, so it hard-pops; and it mounts/unmounts on nearly every keystroke.
  - Fix: Take the ghost out of the centering flow: position it absolutely, anchored to the end of the input (caret position) so its width never moves the operator's text, and keep it always-mounted with an opacity/`content` swap eased over var(--tex-t1) so it fades rather than pops. This is subtraction of jitter, not a new system; grounded completion behavior is untouched.
- **[typing] The line's value is fully controlled, re-rendering the entire 4,300-line Vigil component on every character** — `src/components/Dashboard/Vigil.jsx:4211` (high/day)
  - Symptom: The input feels a hair behind the keyboard, and the lag gets worse the more is happening on the surface (an answer lingering, held rows, spans). It reads as software latency, not a precision instrument.
  - Root cause: value={typed ?? ""} makes the input a controlled component whose onChange calls setTyped(value) (line 2927). Vigil is a single monolith holding dozens of useState/useRef and an enormous JSX tree, so each keystroke reconciles the whole component before the character is confirmed to the DOM. The TYPING_SOTA_BUILDLOG explicitly claimed 'no per-keystroke React re-render of its text' — the evolved controlled design silently violated that parity guarantee.
  - Fix: Decouple the hot path from the monolith: extract the typed line (input + mirror + ghost) into its own memoized child component that owns `typed`/`ghost` locally, so a keystroke re-renders only the line, not all of Vigil; lift to the parent only on submit/cancel. Or make the input uncontrolled (defaultValue + read el.value on submit) and drive only the ghost from state. Behavior identical; keystroke cost drops to the line alone.
- **[typing] Every keystroke forces a synchronous reflow (getBoundingClientRect) before paint to size the input** — `src/components/Dashboard/Vigil.jsx:3014` (high/hours)
  - Symptom: Adds a measurable stall between key press and the character/caret updating, especially on longer questions — part of the 'choppy, not seamless' feel and worse on lower-powered machines.
  - Root cause: The useLayoutEffect runs on every `typed` change and does mirror.getBoundingClientRect().width (a forced layout read) then writes el.style.width — a read-then-write layout thrash executed synchronously before the browser can paint the keystroke. Combined with the full-component re-render (finding #3), each character does render → forced layout → style write → paint.
  - Fix: Eliminate the per-key measurement. Use CSS `field-sizing: content` on .tex-line (progressive enhancement; keep the hidden mirror only as the fallback for engines without it), or render the visible text in the mirror and overlay a fixed-width transparent-text input for caret/entry so there is nothing to measure per key. If measurement must stay, batch it into the ghost's rAF and read before any writes. North-Star target: keystroke→paint in one frame.
- **[typing] First keystroke's caret appears ~2 frames late due to the desktop mount-gap and double rAF** — `src/components/Dashboard/Vigil.jsx:2728` (medium/day)
  - Symptom: The most important moment — the very first letter conjuring the line — is the slowest: the caret and line materialize a beat after you hit the key, and on a fast first word the line briefly shows one character then jumps to the rest. It undercuts the 'telepathic, felt-not-seen' promise.
  - Root cause: On desktop the input does not exist until the first keystroke, so beginTyping sets state, then waits one requestAnimationFrame to focus + drain the mount-gap buffer, then a nested rAF to place the caret (setSelectionRange). That is ≥2 frames before the caret is visible, plus the buffer/replace machinery exists solely to paper over the gap.
  - Fix: Mount the input latent on desktop too (exactly as the touch path already does at Vigil.jsx:4191 — 1px, opacity:0, aria-hidden, pointer-events:none, off the a11y tree), so the first keydown can focus it and insert the char synchronously in the same handler. This collapses first-char latency to one frame AND lets you delete the entire mount-gap buffer (bufferingRef/pendingBufferRef and both rAFs) — a net subtraction. Empty-at-rest is preserved (the latent input is invisible and inert).
- **[typing] First keystroke builds a large SymSpell index that nothing consumes anymore, hitching mid-sentence** — `src/lib/typingAssist.js:75` (medium/hours)
  - Symptom: A stutter can land a few hundred milliseconds into your first typed sentence — right as you get going — as the typing-aid chunk loads and indexes.
  - Root cause: loadAssist (Vigil.jsx:1188) fires on the first keystroke and init() synchronously builds a SymSpell symmetric-delete index (_deletes) over all 9,858 dictionary words — tens of thousands of Map entries via deletes1() — plus the completion trie. That delete index exists only to serve correct(), which is retired and, grep-confirmed, called from nowhere. Only the trie (complete) is actually used. The heavy, dead build runs on the main thread during active typing.
  - Fix: Build only the completion trie in init(); delete the _deletes construction loop and the now-dead correct(), osaDistance(), and deletes1(). This removes the main-thread hitch and shrinks the lazy chunk. Grounded completion and the general trie fallback are untouched.
- **[typing] Blurring the line with text in it strands an unfocused, un-refocusable question — new keystrokes vanish** — `src/components/Dashboard/Vigil.jsx:2913` (medium/hours)
  - Symptom: Type a few words, click elsewhere or alt-tab away and back, then keep typing — nothing appears. The half-written question sits there (invisible, unfocused) and every new keystroke is silently swallowed, with no obvious way to get back into the line.
  - Root cause: onTypedBlur only cancels when the line is empty; a non-empty line stays mounted but loses focus. Meanwhile the document keydown listener early-returns whenever typingRef.current is true (Vigil.jsx:2976), treating 'a line is open' as 'the input owns the keys' — but after a blur the input is not the active element, so the keys land on <body> and are lost. There is no path back to focus.
  - Fix: In the keydown listener, when typingRef.current is true but the input is not focused, refocus and let the key insert — but gate it exactly like the existing at-rest guard (Vigil.jsx:2952-2961): only steal the key when document.activeElement is null/body (or otherwise NOT an INPUT/TEXTAREA/contentEditable/[data-act] element), so a keystroke aimed at a focused held-card button (Refuse/Keep/Seal) or another editable is never hijacked while a line happens to be open.
- **[typing] Mobile line is pinned to layout-viewport center with no on-screen-keyboard handling, so it can sit under the keyboard** — `src/components/Dashboard/Vigil.css:1277` (medium/hours)
  - Symptom: On a phone the composing line and caret can end up crowded against or partly hidden behind the on-screen keyboard, and it never lifts as you type — typing feels cramped and you can't always see what you're writing.
  - Root cause: .tex-line-slot is position:absolute; top:50%; translate(-50%,-50%) — resolved against the full layout viewport, which iOS Safari does NOT shrink when the keyboard opens. The TYPING_SOTA design called for repositioning above the keyboard via VisualViewport resize, but grep confirms no visualViewport usage exists anywhere in Vigil. The line stays at 50% of full height regardless of keyboard occlusion.
  - Fix: Add a VisualViewport 'resize'/'scroll' listener (guarded to coarse-pointer) that offsets the slot to sit just above window.visualViewport.height when the keyboard is up, easing with var(--tex-t3). Purely additive, coarse-pointer only; desktop untouched.

  - Symptom: Ask a question, read the answer, then press/type to ask again — the whole answer block (the spoken line, its credibility tier, the proof pills) vanishes in a single frame. - **[motion] The held card replays its full entrance rise (and two systems fight its opacity) every time an overlay clears off it** — `src/components/Dashboard/Vigil.css:171` (medium/hours)
  - Symptom: After you ask a question over a held decision and then return to silence, the held card doesn't calmly fade back — it re-rises from below and re-fades-in as if it just arrived, a second entrance the operator already saw. It reads as the surface 'redoing' itself.

## Appendix B — July-2026 research recommendations

- **Same-document View Transitions API (document.startViewTransition)** (platform) — A native API that snapshots the old DOM, lets you mutate to the new state in a callback, then cross-fades/morphs between the two automatically. In an SPA you call document.startViewTransition(() => { /* apply state change */ }) and style ::view-transition-* pseudos in CSS. Level 2 adds view-transition-types + the types:[] argument and :active-view-transition-type() so one call can drive different animations (e.g. answer-arriving vs held-card-dismiss), plus auto-naming and nested groups.
  - Why it fits Tex: This is the single biggest lever for the 'one material' morph feel the memory notes describe (VT ~320ms, surface changes ride inside the callback). It replaces hand-written FLIP/opacity choreography and the risk of torn snapshots: ask->answer, held-count->one-at-a-time walk, and seal reveal all become one native transition instead of overlapping JS animations (a known source of the choppy/overlap RCA).
  - Status July 2026: Cross-browser for same-document now: Chrome/Edge 111+, Safari 18+ (WebKit), Firefox 144+ (Gecko). Cross-document (@view-transition) is Chrome 126+/Safari 18.2+, Firefox still in progress behind a flag. Note React's own <ViewTransition> component is still experimental/Canary-only, so use the DOM API directly, not the React component. Sources: developer.mozilla.org/en-US/docs/Web/API/View_Transition_API, web.dev/learn/css/view-transitions-spas, drafts.csswg.org/css-view-transitions-2/
  - First step: Wrap the existing state setter for a UI-mode change (ask->answer, show held) in a helper: `function morph(update){ if(!document.startViewTransition||matchMedia('(prefers-reduced-motion:reduce)').matches){update();return;} document.startViewTransition(()=>flushSync(update)); }`. Use flushSync so React commits inside the callback. Give the answer/seal element `view-transition-name` and tune duration/easing on ::view-transition-group in CSS. No library, feature-detected, degrades to an instant swap.
- **@starting-style + transition-behavior: allow-discrete (+ overlay)** (platform) — Native enter/exit animation primitives. @starting-style declares the 'from' values applied on first render (or when leaving display:none), and transition-behavior: allow-discrete lets discrete properties like display and overlay participate in a transition so an element can animate in AND out without JS mount/unmount timing.
  - Why it fits Tex: Held-decision cards, the count-first summary, toasts, and the passcode dialog can fade/scale in and out purely in CSS, removing the JS timers that currently sequence mount/unmount and contribute to overlap. It is the calm, no-dependency replacement for Framer-Motion-style AnimatePresence in a stark-white UI.
  - Status July 2026: Baseline Newly Available across all engines: Chrome/Edge 117+, Safari 17.5+, Firefox 129+. Animating out from display:none and ::backdrop via overlay works in Chromium and is landing elsewhere; treat exit-from-display:none as progressive enhancement. Sources: web.dev/blog/baseline-entry-animations, developer.mozilla.org/en-US/docs/Web/CSS/@starting-style, developer.mozilla.org/en-US/docs/Web/CSS/transition-behavior
  - First step: On the held-card / toast rule add `transition: opacity .32s, transform .32s, display .32s allow-discrete;` then `@starting-style{ opacity:0; transform:translateY(6px); }` and a `.leaving` (or `display:none`) end state. Delete the corresponding setTimeout-based unmount animation. Guard the whole block behind `@media (prefers-reduced-motion: no-preference)`.
- **linear() easing for springs/bounce (no JS spring library)** (platform) — The linear() timing function interpolates through a list of points, letting a build-time generator bake a real spring/bounce curve (values can overshoot >1) into a single CSS easing value. There is still NO native spring() timing function in shipping CSS as of mid-2026 — linear() is the sanctioned mechanism.
  - Why it fits Tex: Gives physically-plausible settle on the caret, focus ring, seal, and card motion with zero runtime JS, so springs cost nothing on the main thread (protects INP) and never fight the View Transition. Directly removes any need to pull in spring-easing/Motion for micro-interactions.
  - Status July 2026: Baseline / widely available in all engines. The often-repeated 'Safari doesn't support linear()' line is stale — WebKit shipped it in Safari 17.2 (Dec 2023); Chrome/Edge 113+, Firefox 112+. Sources: caniuse.com/mdn-css_types_easing-function_linear-function, web-platform-dx.github.io/web-features-explorer/features/linear-easing/, developer.chrome.com/docs/css-ui/css-linear-easing-function
  - First step: Generate one or two curves with a linear()-easing generator (e.g. a gentle spring and a soft settle), store them as `--ease-spring` / `--ease-settle` CSS custom properties in the root stylesheet, and swap them in for the current cubic-bezier/ease-out on transitions and View Transition pseudos. No dependency added.
- **INP: scheduler.yield() with a setTimeout fallback** (platform) — scheduler.yield() pauses a long task and resumes it as a prioritized (user-visible) continuation, so the browser can paint and process input between chunks. It is the current web.dev-recommended fix for the INP metric (which replaced FID and is the most-failed Core Web Vital in 2026, threshold <200ms).
  - Why it fits Tex: The choppy-UI/overlap RCA is fundamentally main-thread contention. Yielding inside token-processing, markdown parsing, or seal/verify work keeps the caret and hold-to-speak responsive during streaming — the difference between ChatGPT-smooth and janky typing.
  - Status July 2026: Shipping unflagged in Chromium (Chrome/Edge ~129+); NOT yet in Safari or Firefox, so it must be used with a fallback. Progressive-enhancement safe. Sources: developer.chrome.com/blog/use-scheduler-yield, developer.mozilla.org/en-US/docs/Web/API/Scheduler/yield, caniuse.com/mdn-api_scheduler_yield
  - First step: Add a tiny helper `const yieldNow = () => globalThis.scheduler?.yield ? scheduler.yield() : new Promise(r=>setTimeout(r,0));` and `await yieldNow()` inside any loop that processes a full streamed answer, parses long markdown, or renders a batch of held rows. Paint the visible update first, yield, then do non-critical work.
- **Streaming-text render pattern: buffer in a ref, flush on requestAnimationFrame** (platform) — The now-standard architecture for LLM/token streams: the transport writes incoming tokens to a useRef array (outside React state); a single requestAnimationFrame loop reads the buffer, does ONE setState with the accumulated text, and clears it. This decouples the 30-50 events/sec network cadence from React's commit cadence (~60fps, effectively far fewer commits).
  - Why it fits Tex: Per-token setState is the classic cause of streaming jank (commits balloon to 50ms+ in deep trees and block input). Since Tex speaks the answer as it streams while the user may keep typing/holding, rAF-batching is what makes the text arrive smoothly without starving the voice/caret loop.
  - Status July 2026: Pure platform pattern (rAF is universal); no new API required. This is the documented ChatGPT-smooth approach. Sources: developer.chrome.com/docs/ai/render-llm-responses, sitepoint.com/streaming-backends-react-controlling-re-render-chaos/, akashbuilds.com/blog/chatgpt-stream-text-react
  - First step: In the answer component, push SSE/stream chunks into a `bufferRef.current`, start one rAF loop on stream open that does `setText(t => t + bufferRef.current.splice(0).join(''))`, and cancel it on stream end. Fade each newly-committed span in with a short CSS opacity transition rather than animating character-by-character.
- **React useDeferredValue for typing-vs-streaming (keep React 18)** (platform) — useDeferredValue produces a lagging copy of a fast-changing value so the urgent update (the input's own value / caret) commits immediately while the expensive consumer (rendered markdown, filtered exhibit list, streamed answer body) re-renders at lower priority. useTransition does the same for the state update itself and yields an isPending flag.
  - Why it fits Tex: This is the correct React-18 tool for the exact typing-vs-streaming tension in the brief: the type-to-ask field stays perfectly responsive while the heavy answer surface catches up. Rule of thumb from current guidance: useDeferredValue/useTransition are for CPU-bound render work; network calls should still be debounced, not transitioned.
  - Status July 2026: Stable since React 18; unchanged and endorsed in React 19.2 (current stable). No upgrade required. Sources: react.dev/reference/react/useDeferredValue, react.dev/reference/react/useTransition, nutrient.io/blog/react-usetransition-guide/
  - First step: Where the streamed/markdown answer is rendered from a fast-updating string, wrap the consumer with `const deferred = useDeferredValue(answerText)` and render markdown from `deferred`, pairing it with `useMemo` keyed on `deferred` so parsing only reruns when the deferred value settles. Keep the raw value driving the caret/input.
- **React Compiler v1.0 (auto-memoization)** (platform) — A build-time Babel/Oxc transform that automatically memoizes components and values, removing most manual useMemo/useCallback/React.memo. Meta reports ~12% faster initial loads and up to 2.5x faster interactions in production.
  - Why it fits Tex: A voice+typing UI re-renders constantly (streaming text, pulse, timers). The compiler cuts the wasted re-renders that cause micro-jank without you hand-auditing memoization, and it lowers CPU during streaming — directly on-target for 'feel perfectly fluid' with no architectural change.
  - Status July 2026: Stable v1.0 since Oct 7 2025, production-ready, battle-tested at Meta. Supports React 17+, so it works on the current React 18 app (add react-compiler-runtime + a target config for <19). eslint-plugin-react-hooks now ships the compiler lint rules. Sources: react.dev/blog/2025/10/07/react-compiler-1, react.dev/learn/react-compiler/installation
  - First step: `npm i -D babel-plugin-react-compiler@latest` and add it as a Babel plugin inside @vitejs/plugin-react in vite.config; set `target:'18'` and install `react-compiler-runtime`. Turn on the recommended eslint-plugin-react-hooks flat config first, fix any rule-of-hooks violations it surfaces, then enable the compiler. Roll out incrementally via a directory glob; measure, don't rip out existing memoization on day one.
- **content-visibility: auto + contain-intrinsic-size for the transcript** (platform) — content-visibility: auto tells the browser to skip layout, style, and paint for off-screen subtrees; contain-intrinsic-size (or its auto keyword, which remembers the last rendered size) reserves space so the scrollbar stays stable. It is effectively free, CSS-only virtualization.
  - Why it fits Tex: As the conversation trail / held-decision log grows, rendering cost per keystroke and per streamed frame climbs. Marking each past message/row as content-visibility:auto keeps commit and paint cheap regardless of history length, protecting the streaming and typing smoothness without adding react-window.
  - Status July 2026: Baseline: Chrome/Edge 85+, Firefox 125+, Safari 18+. One caveat: Safari's Cmd+F may not find text inside currently-skipped subtrees. Sources: developer.mozilla.org/en-US/docs/Web/CSS/content-visibility, web.dev/articles/content-visibility, caniuse.com/css-content-visibility
  - First step: On the repeated message/row class add `content-visibility:auto; contain-intrinsic-size:auto 120px;` (tune the placeholder to a typical row height). Verify scroll-anchoring stays stable while streaming; if Cmd+F-in-history matters, exempt the last N rows.
- **Web Animations API (Element.animate + commitStyles)** (platform) — A native imperative animation API: element.animate(keyframes, { duration, easing }) returns an Animation you can await, cancel, reverse, or commitStyles() to persist. It accepts the same linear() spring easings as CSS and runs off the JS animation-library path.
  - Why it fits Tex: For the handful of one-off, JS-orchestrated moments that aren't natural CSS state transitions — the seal drawing itself, a focus ring landing, an interrupt/Escape-to-silence flourish — WAAPI gives precise promise-based sequencing and clean cancellation (matching the 'latest-wins ask-epoch' pattern) with zero dependency, so you can retire any residual GSAP/Motion usage.
  - Status July 2026: Mature and Baseline across all engines (including commitStyles and composite modes); not new, but the native answer to 'the platform now does it, drop the library.' Sources: developer.mozilla.org/en-US/docs/Web/API/Web_Animations_API
  - First step: Replace any imperative library tween with `const a = el.animate([...],{duration:320, easing:'var-equivalent linear()', fill:'forwards'})`; cancel the previous Animation when a new ask epoch starts (`a.cancel()`), and `await a.finished` when you need to sequence. Feature-detect prefers-reduced-motion and pass duration:0.
- **interpolate-size: allow-keywords / calc-size() (animate to height:auto)** (platform) — Setting interpolate-size: allow-keywords (usually on :root) lets height/width transitions interpolate between a length and an intrinsic keyword like auto/max-content; calc-size() allows math on intrinsic sizes. It kills the classic scrollHeight/max-content hack for expand/collapse.
  - Why it fits Tex: The held-decision card expanding to reveal detail, or the answer container growing as text streams in, can now grow smoothly to its natural height in pure CSS instead of measured-JS height animation — fewer layout reads, no ResizeObserver dance, calmer motion.
  - Status July 2026: Chromium-only (Chrome/Edge 129+); NOT in Firefox or Safari yet, so strictly progressive enhancement — unsupported engines snap to final height, which is acceptable. Sources: developer.chrome.com/docs/css-ui/animate-to-height-auto, developer.mozilla.org/en-US/docs/Web/CSS/interpolate-size
  - First step: Add `:root{ interpolate-size: allow-keywords; }` and change the expand transition to `transition: height .32s; height:0; ` -> `height:auto;`. Keep the existing collapsed/expanded logic; because it degrades to a snap, no fallback code is required, but verify the snap in Safari looks acceptable.
- **Scroll-driven animations (animation-timeline: view()/scroll())** (platform) — Bind a CSS animation's progress to scroll position or an element's visibility (view progress) with animation-timeline, entirely off the main thread — no scroll listeners, no IntersectionObserver for reveal effects.
  - Why it fits Tex: For a long transcript, subtle reveal/settle of messages as they scroll into view (or a scroll-linked progress cue) stays smooth even while the main thread is busy streaming, because the animation runs on the compositor. It is the calm, jank-proof way to add the little bit of motion a stark-white UI can afford.
  - Status July 2026: Chrome/Edge 115+ (2023), Safari 26+ (Sept 2025, threaded in 26.4). Firefox still behind layout.css.scroll-driven-animations.enabled in stable but on by default in Nightly and an Interop 2026 priority; ~82% global. Progressive enhancement. Sources: webkit.org/blog/17101/a-guide-to-scroll-driven-animations-with-just-css/, developer.mozilla.org/en-US/docs/Web/CSS/animation-timeline, caniuse.com/mdn-css_properties_animation-timeline_scroll
  - First step: For a fade-in-on-enter reveal, give rows `animation: reveal linear both; animation-timeline: view(); animation-range: entry 0% cover 30%;` and set the keyframe from opacity:0/translateY. Keep `animation-duration:1ms` as the Firefox/no-support guard so unsupported engines show the final state. Gate behind prefers-reduced-motion.
- **prefers-reduced-motion as the global gate (calm-by-default)** (platform) — The reduced-motion media query, wired as the master switch over every transition/animation above, so motion is opt-in per the user's OS setting.
  - Why it fits Tex: The product soul is 'silent until approval needed,' stark-white, calm. Motion must never fight that. A single reduced-motion discipline guarantees View Transitions, springs, entry/exit, and scroll reveals all collapse to instant, honest state changes for users who ask for it — and it's an accessibility expectation reviewers check.
  - Status July 2026: Universally supported; stable for years. Source: developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion
  - First step: Add one global rule `@media (prefers-reduced-motion: reduce){ *,*::before,*::after{ animation-duration:.001ms!important; transition-duration:.001ms!important; } }` and mirror the same check in the JS View Transition / WAAPI / scheduler helpers (already shown above) so both layers agree.

  - Why it fits Tex: Not a runtime-fluidity feature, but 10-30x faster builds and faster HMR tighten the iteration loop for all the UI polish work, and smaller/faster tooling is a low-risk win. Adoptable without touching app code.
  - Status July 2026: Vite 7 shipped June 2025; Vite 8 went stable March 12 2026 with Rolldown as the single bundler; Rolldown 1.0 stable May 7 2026. Requires Node 20.19+/22.12+. Mostly drop-in from v7. Sources: vite.dev/blog/announcing-vite8, vite.dev/blog/announcing-vite7, github.com/vitejs/vite/releases

- **CSS reading-flow / reading-order (focus-order correctness)** (platform) — reading-flow lets a flex/grid container expose its children to tab order and assistive tech in a logical sequence, and reading-order nudges individual items, so visual reordering no longer desyncs keyboard/AT navigation.
